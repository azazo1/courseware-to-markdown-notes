# 课件转 Markdown 笔记

将课件逐页视为图片来阅读, 清点每一页的完整内容, 然后重建为适合初学者阅读的优雅 Markdown 笔记. 将视觉检查视为主要证据来源, 不要用 OCR, PDF 文本提取, 或直接复制幻灯片文字来替代阅读.

默认使用单 agent 路径完成整套课件转换. 只有用户明确要求使用多个 sub-agent, 或明确要求分块并行处理时, 才启用 sub-agent 路径. 分块只是为了控制图片上下文负载, 不是为了降低覆盖率, 讲解性, 或成稿质量.

## 快速开始

1. 先将材料规范化为页面图片.
   - 对于 `.pdf`, 运行 `uv run python scripts/render_courseware_pages.py <input.pdf> --output-dir <scratch-dir>`.
   - 默认输出为 `webp` 页面图片, 便于后续直接目视阅读.
   - 对于幻灯片图片目录, 也用同一个脚本处理整个目录.
   - 对于 `.pptx` 之类暂不支持的格式, 先导出为 PDF, 或请用户提供 PDF 或图片导出结果. 不要把 OCR 当作替代方案.
2. 创建逐页记录用的临时模板.
   - 运行 `uv run python scripts/init_page_capture_template.py <scratch-dir>/manifest.json --output <scratch-dir>/page-capture.md`.
3. 默认不创建分块计划.
   - 单 agent 路径下, 直接按页面顺序阅读, 记录, 写入, 修订.
   - 只有用户明确要求使用多个 sub-agent 或分块并行时, 才运行 `uv run python scripts/split_manifest_chunks.py <scratch-dir>/manifest.json --output <scratch-dir>/chunks.json`.
   - 如果启用 sub-agent, 大多数情况下每块约 10 页比较稳定. 如果页面密度很高, 可以缩小到 6 到 8 页.
4. 尽早创建最终笔记文件.
   - 一旦能确认局部主题, 就创建目标 Markdown 文件, 先写入当前已经读明白的内容和临时小节.
   - 不要在读完整套材料前强行预设完整章节骨架. 逐页目视阅读时, 允许 heading 先保持临时状态, 再按输出约定逐步整理.
5. 逐页目视阅读所有渲染后的页面.
   - 直接检查页面图片.
   - 逐页填写临时模板, 确保没有任何页面被跳过, 但要把它当作覆盖情况台账, 而不是逐页摘要表.
   - 如果启用 sub-agent, 按块派发页面图片, 再由主 agent 统稿.
6. 逐步写入最终 Markdown 笔记.
   - 每完成一个相对稳定的局部主题, 小节, 或页面组, 就立刻把这一部分写进最终笔记.
   - 你最多连续读取 10 张图片而不进行写入, 超过则必须先将当前的内容写入笔记, 然后再继续读取图片.
   - 随着理解加深持续修订已有内容, 不要一直等到最后再一次性写完.
   - 优先还原课件中的表达方式与组织形式, 不要为了统一风格而过度改写.
7. 对照 [references/output-contract.md](references/output-contract.md), [references/latex-and-tables.md](references/latex-and-tables.md), 和 [references/subagent-chunk-contract.md](references/subagent-chunk-contract.md) 复查最终笔记与分块交付质量.

## 必需工作流

### 0. 选择执行路径

- 默认走单 agent 路径. 主 agent 自己负责渲染, 逐页阅读, 记录, 写入最终笔记, 修订 heading, 和全局完整性复查.
- 只有用户明确要求使用多个 sub-agent, 分块, 并行, 或类似说法时, 才走 sub-agent 路径.
- 无论走哪条路径, 最终笔记都必须满足这份 skill 与 [references/output-contract.md](references/output-contract.md) 的全部要求.

### 0.1 如果使用多个 sub-agent

- 主 agent 负责渲染, 切块, 逐步落地局部内容, 派发任务, 整合分块结果, 修订终稿, 以及做全局完整性复查.
- 子 agent 只负责自己那一块连续页面图片的阅读, 记录, 局部成稿, 和疑点上报. 子 agent 不是最终统稿者, 也不负责整套讲义的全局结构裁决.
- 派发给子 agent 时, 必须把当前 skill 一并传给子 agent. 如果工具支持 `skill` item, 直接传入当前 skill, 不要只发一段简化口头要求.
- 同时明确告诉子 agent:
  - 你是子 agent, 只负责指定页块.
  - 你必须遵循本 skill 的全部内容质量要求, 但只对本块内容负责.
  - 你不能因为自己是子 agent, 就把输出退化成逐页摘要或低密度摘录.
- 主 agent 不要把完整对话历史, 整套图片, 全部已完成章节, 或所有块的详细结果都塞给子 agent. 只传当前块需要的最小上下文.
- 主 agent 在派发后不要继续重复读取同一批图片, 以免主上下文再次被图片撑爆. 只有在子 agent 报告冲突, 歧义, 或明显缺失时, 才回头抽查个别页面.
- 每个子 agent 处理 6 到 10 页, 大多数情况下约 10 页比较稳妥. 页面极密时主动缩小块大小.
- 子 agent 的交付格式遵循 [references/subagent-chunk-contract.md](references/subagent-chunk-contract.md).
- 主 agent 只能把分块读图委派出去, 不能把最终质量标准一并委派掉. 最终笔记仍然必须满足这份 skill 与 [references/output-contract.md](references/output-contract.md) 的全部要求.

### 1. 先渲染, 后阅读

- 在解读内容之前, 先把源材料转换成有序的页面图片.
- 将渲染后的页面图片作为文本, 图形, 图表, 表格, 公式, 高亮, 以及版式重点的主要事实来源.
- 如果源材料本身已经是图片集, 也要用 `scripts/render_courseware_pages.py` 再规范化一次, 让页面顺序统一, 并记录到同一个 manifest 中.

### 2. 在改写前逐页记录

- 使用 `scripts/init_page_capture_template.py` 创建一个列出全部页面的工作文档.
- 对每一页, 记录以下内容:
  - 这一页在当前章节或论述中的作用
  - 转换后必须保留的标题, 小标题, 列表, 提示框, 内容块
  - 重要定义, 结论, 阈值, 示例
  - 具有实际意义的表格, 示意图, 图表, 截图, 版式关系
  - 用 LaTeX 改写后的公式与符号
  - 警告, 限制条件, 边界情况
- 标题页, 议程页, 章节分隔页, 回顾页也都算有意义的内容. 它们往往在说明原始结构.
- 如果使用子 agent, 上述逐页记录要求同样适用于子 agent. 只是记录范围缩小到当前块, 不代表记录密度可以降低.

### 3. 将课件重建为 Markdown

- 将材料重建为读起来自然的 Markdown 笔记, 深入浅出地讲解其中的知识概念, 帮助初学者入门, 保留所有有意义的教学内容.
- 除非为了可读性需要做少量局部调整, 否则保持课件原有的逻辑顺序.
- 保留标题, 小标题, 编号步骤, 嵌套列表, 定义, 示例, 例题, 操作流程, 对比, 警告, 结论.
- heading 结构可以在阅读过程中逐步形成. 定稿时按 [references/output-contract.md](references/output-contract.md) 统一 heading 层级, 术语风格, 页码引用, 覆盖完整性, 和可读性.
- 最终成文不应只是把原页内容按顺序重新排成清单. 要围绕概念, 原理, 条件, 推导, 结论之间的关系来组织表达.
- 如果多页重复同一个观点, 只合并重复部分. 不要把不同的例子, 例外情况, 支撑细节一并压缩掉.
- 将逐页记录文件当作完整性检查工具, 不要把它直接当作最终叙述形态.
- 只要一个局部主题或页面组已经足够清楚, 就开始写最终笔记.
- 优先采用按局部主题逐步推进的写法, 不要等到最后再进行一次性的大规模生成.
- 如果后续页面补充了必要上下文, 要回头修订前文, 但不要把所有写作都拖到最后, 以免信息已经被过度压缩.
- 如果使用子 agent, 子 agent 应尽量交付可直接并入终稿的局部草稿, 而不是仅交付松散摘录. 主 agent 再负责跨块整合与术语统一.

### 4. 面向初学者写作

- 术语第一次出现时就给出定义, 并补充稳定的中文对应翻译.
- 默认使用中文教材式表达. 专有名词, 缩写, 或英文技术术语首次出现时做中英对照, 后文以中文名称或中文说明为主.
- 先给简短解释, 再给正式表达.
- 如果某一页隐含了背景知识, 补上一句必要上下文, 让整份笔记自洽.
- 如果某个概念, 公式, 图示, 或论证第一眼不容易看懂, 补一小段白话解释.
- 这些解释应当简短且局部, 只澄清难点, 不要把笔记扩写成整章教材.
- 让笔记既完整, 也好读.
  - 与其机械地拼成长段, 不如优先使用聚焦单一主题的短段落.
  - 在有帮助的地方补一两句过渡或引入, 让读者更容易跟上思路.
  - 如果某一节信息密度很高, 可按需要拆成 定义, 机制, 步骤, 例子, 说明, 注意, 结论 等小节.
  - 并列内容用列表, 过程内容用有序列表, 结构化对比用表格, 不要全部压成散文.
  - 在密集公式块, 对比表, 或推导后面, 补一句简短总结, 点明主要结论.
- 优先采用深入浅出的讲解方式.
  - 先说明这个知识点在解决什么问题, 再展开定义, 机制, 条件, 结论.
  - 如果一组知识点存在因果, 并列, 层级, 或前后依赖关系, 要把这种关系讲清楚, 不要只是并排列出.
  - 能用一两句自然语言解释清楚的地方, 不要退化成生硬的关键词堆砌.
- 默认读者看不到原始课件, 只能看到这份 Markdown 笔记.
- 让笔记本身可以独立成立. 不要让读者回看幻灯片补信息.
- 使用直接陈述知识的笔记口吻, 而不是课件摘要口吻.
- 写内容本身, 不要描述幻灯片或课件在做什么.
- 将碎片化的幻灯片语句改写成完整句.
- 优先使用清晰的 Markdown 结构, 不要保留幻灯片版式噪音, 但要保留原有分组和强调的意义.
- 页码引用应放在标题下方的引用块中, 不要直接写进标题文本.

## 输出规则

- 覆盖输入讲义中的全部页面. 任何页面都不能无声消失.
- 除非用户明确要求其他格式, 否则最终产物保持为 Markdown.
- 除非用户明确要求其他语言, 否则默认用中文写笔记.
- 最终产物应当读起来像打磨过的课程笔记或讲义, 而不是逐页摘要列表.
- 最终产物应当便于浏览和复习, 并在定稿时满足 [references/output-contract.md](references/output-contract.md) 的全部检查项.
- 在阅读过程中就持续增量写入最终笔记, 不要等所有页面都处理完再动笔.
- 优先尽早落地已经完成的局部内容, 让整份笔记在全部完成之前也具备可用性.
- 在中文正文中遵循盘古之白.
  - 在中文与相邻英文单词, 数字, 以及多数行内公式之间酌情加入空格, 让阅读更自然.
  - 但不要加入会破坏 Markdown 语法, 行内代码, 链接, 或 LaTeX 定界符的空格.
- 将所有有语义的课件内容转换成 Markdown 结构. heading 层级, 列表, 表格, 引用块, 和页码追溯按输出约定统一处理.
- 公式和特殊符号统一使用 LaTeX.
  - 短表达式使用行内公式, 例如 `$p(x)$`.
  - 多步公式或推导使用块级公式.
  - 不要用反引号表示公式, 变量, 不等式, 方程, 矩阵, 或符号表达式.
  - 反引号只用于真正的代码, 文件路径, 命令, 字面语法, 或标识符.
  - 当存在可用的 LaTeX 命令时, 不要保留原始 Unicode 数学符号.
- 如果一句话中包含数学关系, 应改写到数学模式中, 不要保留为普通文本或行内代码.
- 公式内部和周围的标点也应保持半角.
- 当中文正文包围行内公式时, 在公式定界符外应用盘古之白, 例如 `设 $x > 0$, 则函数在 section 2 中单调递增.`
- 重要表格要转换为 Markdown 表格.
- 当图表, 示意图, 截图本身承载了纯文字无法表达的含义时, 要用文字解释其作用.
- 对于直接转写后仍然会让初学者困惑的隐含内容或难点, 需要做简短说明.
- 保留那些对理解概念, 方法, 条件, 边界, 或结论有直接帮助的例子和使用场景.
- 如果某些事件, 背景故事, 应用轶事, 历史插曲, 或展示性材料与知识讲解关系不大, 可以直接省去.
- 除非用户明确要求逐页摘要, 否则不要让最终笔记变成逐页流水账.
- 保留页码或页码范围用于追溯, 但只能作为引用, 不能替代解释.
- 不要写出 `课件里提到`, `见上图`, `如图所示`, `参考第 12 页` 之类默认读者能看到原课件的措辞.
- 不要把整份笔记写成对课件本身的评论.
  - 避免使用 `这组课件围绕...展开`, `这组课件的目标是...`, `课件把...分成...`, `这一页介绍了...` 之类表达.
  - 优先改写成 `...分成 6 类`, `本节讨论...`, `核心问题是...` 这类直接陈述.
- 对于有意义的格式差异, 要翻译成 Markdown 结构, 而不是直接丢弃.
  - 并排对比可以转换为表格
  - 被强调的警告可以转换为引用块或简短警示段落
  - 推导步骤可以转换为对齐的块级公式加简短说明
- 使用的名词尽量和课件中的名词对应, 比如课件中使用 Phase, 不要擅自改成 Pass, 如果课件中某个指标从 1 开始, 不要擅自改成从 0 开始.
- 如果使用子 agent, 分块结果同样必须满足以上全部规则. 子 agent 的分块交付只是最终笔记的一部分, 不能降低最终标准.

## 决策规则

- 如果讲义本身已经有清晰的章节边界, 就在笔记中镜像这种结构.
- 如果源材料结构已经很好, 就保留它, 不要强行发明新的分类体系.
- 如果讲义以流程为主, 就按工作流步骤组织内容.
- 如果讲义以概念为主, 就按概念, 公式, 例子, 结论组织内容.
- 如果同一部分内容既有知识点枚举, 又有内在逻辑, 优先重建逻辑链条, 不要保留机械的罗列顺序.
- 如果某一页以视觉内容为主, 要解释这个视觉材料到底在说明或证明什么.
- 如果某一页包含信息密集的表格, 先在 Markdown 中保留表格结构, 再在下方补一句或几句总结对比结果.
- 如果某个公式只以图片形式出现, 要用 LaTeX 干净地重写出来, 不要含糊描述.
- 如果幻灯片里出现代码, 只有在它确实是代码时才保留为代码块. 数学符号必须留在 LaTeX 中, 不能变成行内代码.
- 如果原文技术上正确, 但压缩得过于厉害, 不利于初学者理解, 就在保留内容的前提下补一句简短解释, 不要直接把难点删平.
- 如果原幻灯片依赖箭头指向, 局部标注, 或极简标签, 就把真正要表达的内容改写完整, 不要回指幻灯片.
- 如果一句话本质上是在描述课件如何呈现内容, 而不是在陈述知识本身, 就把它改写成直接的学科内容表述.
- 如果某一节需要页码追溯, 把追溯信息放到标题下方的引用块, 不要写进标题.
- 如果某一节开始显得过于拥挤, 就先拆成更小的小节, 再继续补内容.
- 如果多个项目符号其实都在解释同一个机制, 就合并成一小段正文加一个简短列表, 不要保留原样的散乱项目堆积.
- 如果某一页包含对比, 流水线, 检查清单, 或分类体系, 优先改成读者一眼能扫清的表格或结构化列表.
- 如果某个例子只是为了活跃课堂气氛, 展示新闻事件, 或罗列应用场景, 但对知识理解没有实质帮助, 可以不写入最终笔记. 但是不是所有的例子都要删去, 比如一个知识点的例题或者类似性质的内容可以适当地保留.
- 如果使用子 agent, 主 agent 在整合时要优先修复跨块重复, 术语不一致, 章节边界分裂, 和页码范围断裂等问题. 不要把分块边界原样暴露到最终笔记里.

## 资源

### `scripts/render_courseware_pages.py`

将 PDF 或幻灯片图片目录渲染为规范化的 `webp` 页面图片, 并生成 `manifest.json`.

### `scripts/init_page_capture_template.py`

生成一个逐页的临时 Markdown 模板, 让你在撰写最终笔记前检查覆盖完整性.

### `scripts/split_manifest_chunks.py`

根据 `manifest.json` 生成适合派发给子 agent 的连续页块计划, 帮助主 agent 控制单次图片上下文规模.

### `references/output-contract.md`

在定稿前用它检查完整性和对初学者的可读性.

### `references/latex-and-tables.md`

在重写公式, 符号, 矩阵, 或表格时参考这个文件.

### `references/subagent-chunk-contract.md`

在使用多个 sub-agent 时, 用它约束子 agent 的角色, 输入范围, 输出格式, 以及主 agent 的派发方式.
